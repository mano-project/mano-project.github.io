Documentation

<MANO> is a lightweight, open digital platform for manuscript studies. It combines a structured metadata editor, a public collection interface, a transcription viewer for TEI-XML, a PAGE-XML → TEI-XML converter, geospatial visualizations, and a curated repository of learning materials. This documentation presents the platform’s goals, data model, validation logic, and technical architecture to support reproducible research, teaching, and responsible data reuse.

<MANO> is conceived as an academic yet accessible infrastructure for creating and disseminating manuscript descriptions and transcriptions. It adheres to established community standards—most prominently the TEI P5 Manuscript Description—while privileging transparency and reuse. The platform is intentionally lightweight: it runs on GitHub Pages, minimizes server complexity, and exposes human-readable files (JSON, TEI-XML) suitable for long-term preservation and scholarly exchange.

Core design principles include: (i) openness (data and code repositories are public); (ii) parsimony (limited but robust feature set); (iii) interoperability (structured exports and references to external authorities); and (iv) pedagogical value (clarity of interface and documentation to support teaching).

  1. Metadata creation in the Metadata Editor.
  2. Validation using built-in checks; revise until no blocking errors remain.
  3. Export JSON (primary) and TEI-XML (for interchange); retain local copies.
  4. Contribution of JSON records to the public repository (see Metadata Collection).
  5. Discovery and reuse via the Collection interface, Maps, and the Transcription Viewer.

Prerequisites: no server access is required. A GitHub account is recommended for contributing records or resources.

MANO maintains a pragmatic JSON schema that maps cleanly to TEI-XML for interchange. Records are organized into the following domains:

  • File Description — file name, language, license, responsible persons (with optional identifiers and affiliations).
  • Identification — manuscript name, repository, settlement, country, shelfmark/sigle, identifiers.
  • Content — items (msItems) with author, title, foliation/page ranges, incipit/explicit, genre, language, and textual family.
  • Physical Description — material, dimensions, leaves/columns/lines, number and description of hands, scripts, decoration, bindings, seals.
  • History — origin (human-readable and machine-readable dates), provenance events with owners, places, and date ranges.
  • Surrogates & Literature — digital copy, IIIF manifest, catalogue links; bibliography with authors/editors and references.

Identifiers and external references. Where possible, fields accept both a display value and a URI (e.g., VIAF, GND, Wikidata, ORCID) to encourage authority control.

TEI export. The editor generates a TEI P5-compliant manuscript description with conservative encoding of the above domains, facilitating reuse in scholarly pipelines and catalogues.

Normative reference. See TEI P5, chapter “Manuscript Description”: https://tei-c.org/.../MS.html

Validation combines mandatory checks (blocking) and optional heuristics (advisory). Examples include:

  • Mandatory — presence of core fields (file/language/license/responsible persons; manuscript name; repository), consistent page-range entries, and chronological coherence of origin and provenance ranges.
  • Optional — realistic dimensions; potential conflation of responsible persons with content authors; matching or suspiciously identical dates between origin and ownership; human-readable vs machine-readable date parity.

Validation is reported within the editor; downloads can be blocked until mandatory errors are resolved. Optional notices promote best practice without preventing export.

Rationale: clarity for students and contributors; defensible, auditable records for scholarly reuse.

Purpose. Create structured manuscript descriptions with immediate feedback.

Key features.

  • Multi-manuscript sessions with add/remove; resume work by re-loading JSON/TEI when compatible with the MANO schema.
  • Contextual validation (mandatory/optional), and download of JSON (canonical) and TEI-XML (interchange).
  • Authority-friendly fields (value + URI) and structured literature entries.

Export recommendations.

  • Retain both JSON and TEI-XML for provenance and citation; include the license and responsible persons.
  • When contributing to the Collection, ensure that human-readable and machine-readable dates are both present where applicable.

The Collection aggregates contributed JSON records and exposes them for browsing, filtering, and download. Each row represents a manuscript; a modal provides a structured view of the record, including links to digital surrogates and catalogue entries where present.

Contribution workflow.

  1. Export JSON from the Editor.
  2. Submit the file to the public repository via pull request or email to the project maintainer.
  3. Upon acceptance, the record appears automatically in the Collection.

Licensing. Contributors must ensure that records are released under the platform’s stated Creative Commons license and observe rights of referenced resources.

Transcription Viewer. Visualizes TEI-XML transcriptions in parallel: code, rendered text, and—when available—digitized facsimiles. The view is synchronized and can be refreshed after in-browser edits.

Converter. When transcriptions exist as multiple PAGE-XML files, the converter merges them into a single TEI-XML suitable for the Viewer. This is convenient for teaching and for integrating external OCR/HTR pipelines.

Good practice. Keep transcriptions and images under clear licenses; cite repositories and shelfmarks; align page breaks (<pb/>) consistently.

The Maps module displays the current place of storage (repository) and places of origin using Leaflet. Each marker aggregates manuscripts associated with the location and links back to the Collection for inspection.

  • Data sources. Repository and origin fields with optional coordinates; countries and settlements may include authority URIs.
  • Interpretation. Points reflect discrete events/places; they are not probabilistic reconstructions.
  • Pedagogy. Supports spatial reasoning about production and transmission.

Hosting. Static site on GitHub Pages; no server-side dependencies.

Client stack. HTML5, Bootstrap 5, vanilla JavaScript; Leaflet for mapping; CodeMirror for XML editing; Font Awesome and Bootstrap Icons for UI.

Data flow. JSON and TEI-XML are produced in-browser; the Collection fetches JSON from the public repository via the GitHub API and renders it into a table and modals; Maps consume the same records to plot locations.

SEO/Structured data. Pages include canonical URLs, Open Graph metadata, and JSON‑LD to improve discoverability and scholarly attribution.

Versioning. Git-based workflows allow transparent revision history and reproducibility; contributors should use descriptive commit messages and PR descriptions.

Governance. The platform is maintained by the project author; contributions are welcomed through public repositories. Issues and proposals should be filed via GitHub for transparency.

Licensing. Unless otherwise noted, site content and contributed metadata are released under Creative Commons Attribution–NonCommercial–ShareAlike (CC BY‑NC‑SA 4.0). External resources referenced from institutions remain under their respective licenses.

Attribution. Cite responsible persons as listed in each record; retain links to authority files and catalogue entries.

Ethics and sensitivity. Where records concern sensitive materials, contributors should respect institutional guidelines and cultural protocols; include notices if access is restricted.
